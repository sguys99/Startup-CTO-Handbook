# 아키텍처

다양한 아키텍처 패턴을 깊이 있게 다루는 훌륭한 자료가 많다. 그중 내가 특히 아끼는 책은 마틴 파울러(Martin Fowler)의 *Patterns of Enterprise Application Architecture*다. 이 장에서는 자주 듣게 될 핵심 용어 몇 가지를 정리한다. 나중에 다른 자료로 이 주제들을 깊이 파고들 때 맥락을 잡는 데 도움이 될 것이다.

## 도메인 주도 설계

도메인 주도 설계(domain-driven design, DDD)는 더 나은 소프트웨어 해법을 설계하기 위해 문제 도메인을 이해하고 모델링하는 데 초점을 맞춘 소프트웨어 개발 접근법이다.

DDD의 핵심 개념은 다음과 같다.

**도메인 모델(domain model):** 비즈니스 개념을 기술 시스템 안의 객체로 표현한 것.

**유비쿼터스 언어(ubiquitous language):** 혼란을 줄이기 위해 회사 전체에서 공통으로 일관되게 쓰는 어휘와 언어.

**경계 컨텍스트(bounded context):** 해당 도메인 모델이 적용되고 유비쿼터스 언어가 통용되는 경계.


### 상위 수준 패턴

누군가 기술 아키텍처(technical architecture)라는 말을 쓸 때, 그것은 대개 코드가 어떻게 실행되고 정보가 시스템 안에서 어떻게 흘러 다니는지를 가리킨다. 아키텍처를 설명할 때는 보통 서비스(service), 모놀리스(monolith), 메시지 전송(message transport) 같은 표현이 등장한다. 이는 객체지향(object-oriented), 함수형 프로그래밍(functional programming), 의존성 주입(dependency injection) 같은 표현이 자주 나오는 코딩 패턴(coding pattern)과는 구분된다. 코딩 패턴은 코드 아키텍처(code architecture)라고 부르기도 하며, [코딩 패턴](#코딩-패턴) 절에서 다룬다.

기술 아키텍처에서 가장 영향이 큰 결정은 코드를 모놀리스로 실행할지, 아니면 여러 서비스(흔히 마이크로서비스라고 부른다)의 집합으로 실행할지다. 먼저 두 패턴이 각각 어떤 모습인지 설명하고, 그다음 둘 사이의 트레이드오프에 관한 지침을 제시하겠다.

#### 모놀리식 아키텍처

모놀리식 아키텍처(monolithic architecture) 패턴은 모든 코드가 하나의 프로세스로 실행되고, 시스템 각 부분 사이의 정보가 전적으로 메모리 안에서 단순한 함수 호출로 오가는 방식이다. 어느 날 오후에 자리를 잡고 간단한 애플리케이션을 만들어 본 적이 있다면, 그것은 십중팔구 모놀리스에 든다. 모놀리스는 아주 작은 것부터 수백만 줄에 이르는 거대한 프로젝트까지 온갖 형태와 크기로 존재한다.

성공적인 모놀리스를 만드는 핵심은 도메인 주도 설계를 활용해 애플리케이션 내부의 데이터 흐름을 세심하게 설계하는 것이다. 잘 됐는지는 꽤 쉽게 가늠할 수 있다. 개발자가 애플리케이션의 기능을 바꾸려 할 때, 모놀리스의 어느 부분을 손봐야 하는지가 분명해야 한다. 목표를 이루려고 손대야 할 코드가 명확하고 잘 정의된, 한정된 영역에만 있어야 한다는 뜻이다. 하나의 기능 요구사항을 충족하느라 코드베이스를 추가로 건드려야 할 때마다 복잡성이 늘고 오류 가능성이 커지며, 대체로 개발 속도가 느려진다.

모놀리스의 핵심 특징:

* 코드가 하나의 단위로 배포된다.
* 코드를 하나의 소스 코드 저장소에서 관리한다.
* 배포된 코드가 하나의 단위로 함께 확장·축소된다.
* 시스템 각 부분 사이의 정보가 메모리 안에서, 보통 함수 호출로 오간다.
* 도메인 주도 설계와 명확한 정보 흐름 설계를 시스템이 강제하지 않으므로, 설계를 잘하는 일은 엔지니어의 몫으로 남는다.

#### 서비스 지향 아키텍처(SOA)/마이크로서비스

서비스 지향 아키텍처(service-oriented architecture, SOA)라는 말은 1990년대에 등장했고, 처음에는 꽤 구체적인 기술 선택을 가리켰다. 요즘은 시스템 각 부분 사이에서 정보가 네트워크를 통해 오가는 시스템을 더 넓게 가리키는 말로 쓰인다. SOA의 가장 큰 트레이드오프는, 모놀리스에 비해 머릿속으로 따져 보기가 매우 복잡할 수 있다는 점이다. 늘어난 복잡성을 상쇄할 만큼 이점이 확실하려면 팀이 상당한 준비와 사려 깊은 설계를 해야 한다.

마이크로서비스(microservice)는 서비스 지향 아키텍처의 한 갈래로, 이름 그대로 각 서비스가 아주 작다. 수천 개의 마이크로서비스로 이뤄진 시스템 구현도 있는데, 각 서비스가 코드 몇 줄에 불과하기도 하다. 그렇다고 서비스 지향 아키텍처의 이점을 누리려고 마이크로서비스가 수천 개씩 필요한 것은 아니다. 적절한 상황이라면 시스템을 더 작은 서비스 네다섯 개로 쪼개는 것만으로도 코드 건전성이 크게 좋아질 수 있다.

마이크로서비스가 유일하게 좋은 아키텍처 패턴이라는 말을 들어 봤을지도 모르겠다. 사실이 아니다. 그런 인식은 많은 모놀리스가 형편없이 설계됐거나, 생산성을 끌어올리는 데 필요한 만큼 기술 부채에 관심과 투자를 받지 못한 데서 비롯한다. 모든 마이크로서비스 아키텍처가 즐겁게 일할 수 있는 환경이라는 생각 역시 사실이 아니다. 이런저런 이유로 그 이점을 제대로 살리지 못하는 마이크로서비스 구현도 많다.

SOA 또는 마이크로서비스 시스템의 핵심 특징:

* 서로 다른 서비스를 독립적으로 배포하고 확장할 수 있다.
* 코드를 하나의 소스 코드 저장소에서 관리할 수도 있고, 여러 저장소로 나눠 관리할 수도 있다.
* 시스템 각 부분 사이의 정보가 네트워크를 통해, 흔히 HTTP·RPC(원격 프로시저 호출, Remote Procedure Call)·큐잉 시스템으로 오간다.
* 계약(contract)이 API로 구현되어 네트워크 너머로 통신되므로, 데이터 계약을 의도적으로 잘 설계해 두어야 한다.

#### 서비스 지향 아키텍처와 모놀리스 사이에서 고르기

일반적으로 모놀리스는 SOA보다 구축하기 쉽고, 관리에 드는 기술적 잡무도 훨씬 적다. 그래서 대다수 문제에서 첫날의 정답은 모놀리스다. 팀이 모놀리스를 설계할 때 충분히 규율 있고 사려 깊다면, 모놀리스는 팀과 함께 끝없이 확장될 수 있다. 다만 모두가 그럴 수 있는 것은 아니다. 많은 팀과 프로젝트에서는 모놀리스가 계약을 강제하지 못하고, 구성 요소별로 따로 확장할 수 없으며, 관심사 분리를 강제하지 못한다는 점이 결국 생산성의 걸림돌이 된다.

다루기 힘든 모놀리스와 씨름하게 됐다고 해서, 엔지니어들이 일을 못한다는 뜻은 아니다. 요구사항이 바뀌고 시스템이 진화하는 것은 소프트웨어 엔지니어링의 본질이다. 모놀리스를 유지한다는 것은 때로 시스템 설계를 함께 진화시키려고 상당한 자원을 들이는 일을 뜻하기도 한다. 모놀리스의 복잡성이 생산성의 걸림돌이 되는 것은, 바로 팀이 이 투자를 게을리할 때다.

서비스 지향 아키텍처로 옮겨 가는 것이 명백히 옳은 선택인 상황도 있다.

서비스에 독립적으로 확장해야 하는 요소가 있는 경우다. 예를 들어 어떤 기능이 CPU 자원을 많이 잡아먹는데 그것이 다른 기능에 지장을 주지 않게 하고 싶거나, 그 한 부분만 따로 확장하는 편이 비용 효율적인데도 모든 기능을 함께 키우는 비용을 치르고 싶지 않은 경우다.

자체적인 독립 API를 노출해야 하고, 메인 시스템과 별개로 고유한 데이터 도메인을 가진 기능을 다루는 경우다. 특히 그 API가 외부 고객을 위한 것이라면, 이 기능을 별도 서비스로 두는 것이 누가 봐도 좋은 선택이다.

어떤 이유로든 애플리케이션의 일부에서 다른 프로그래밍 언어를 써야 하는 경우다. 좋은 예로, 어떤 종류의 문제를 푸는 데 견고하고 품질 좋은 파이썬 프레임워크가 있는데 나머지 애플리케이션은 자바로 되어 있는 상황을 들 수 있다. 두 언어를 메모리 안에서 잇는 것도 가능하지만 영 매끄럽지 않다. 더 쉬운 방법은 API로 잇고, 둘을 자연스럽게 별도 서비스로 띄워 두는 것이다.

모놀리스를 배포하는 일이 지나치게 비싸거나 느리거나 위험한 경우다. 이때는 새 코드를 독립 서비스로 배포해 생산성을 끌어올리고 배포 시간을 줄일 수 있다. 다만 그 새 서비스가 모놀리스와 독립적으로 동작하도록 해서, 새로운 배포 의존성을 만들지 않도록 주의하라.

#### 서비스 지향 아키텍처의 소스 관리: 모노레포와 매니레포

모놀리스의 소스 코드 관리는 비교적 단순하다. 단일 빌드 시스템을 갖춘 하나의 저장소에 코드가 들어 있기 때문이다. 그런데 코드를 여러 패키지·프로젝트·서비스로 쪼개기 시작하면 결정해야 할 문제가 생긴다. 여러 서비스를 하나의 코드 저장소에서 관리할 것인가, 아니면 저장소를 여러 개로 나눌 것인가? 이 트레이드오프를 모노레포(monorepo) 대 매니레포(manyrepo)라고 부른다.

여러 서비스를 모노레포로 관리하기로 했다면, 프로젝트들을 따로 빌드할 수 있도록 워크스페이스 관리 솔루션(예: 자바스크립트 생태계의 yarn workspaces)을 찾아보는 것이 좋다. 모노레포와 매니레포 방식의 기본적인 차이는 다음과 같다.

모든 서비스·패키지 의존성을 최신 버전으로 맞춰 두기가 쉽다.

많은 CI 시스템이 하나의 저장소 안에 여러 패키지가 있는 구조를 기본으로 지원하지 않으므로, 이를 받쳐 줄 하니스(harness)를 직접 만들어야 한다.

**매니레포는 이와 대조적으로**

버전 관리가 되는 중앙 패키지 매니저를 써야 한다. 이것이 반드시 나쁜 것은 아니지만, 한 프로젝트와 그 의존성을 동시에 작업할 때 상당한 부담이 될 수 있다.

CI/CD 파이프라인 시스템(Bitbucket pipelines, GitHub actions 등)과 깔끔하게 통합된다.

내 일반적인 조언은, 단순하게 가라는 것이다. 소규모에서 중간 규모의 프로젝트라면 모노레포가 구축과 유지보수 모두 더 간단하다. 매니레포로 전환한다는 것은 개발자들이 매니레포를 매끄럽게 쓸 수 있도록 도구에 기꺼이 투자하겠다는 뜻이고, 그 비용은 만만치 않다. 작은 스타트업이라면 그 비용을 치를 가치가 없을 가능성이 크다. 반대로 빠르게 성장 중이거나 개발자가 쉰 명을 넘어서고 있고, 모노레포가 점점 다루기 버거워지는데, 매니레포를 쓰기 편하게 만드는 궂은일을 맡아 줄 전담 내부 플랫폼 팀이나 DevOps 팀이 있다면, 매니레포 패턴으로 전환하는 것이 옳은 선택일 수 있다.

#### 분산 모놀리스

분산 모놀리스(distributed monolith)는 여러 서비스로 배포되어 있지만 충분한 독립성이나 격리를 갖추도록 설계되지 않아, 결국 독립적으로 배포할 수 없는 시스템이다. 분명히 말하지만 이것은 두 세계의 단점만 모은 것이다. 개발자가 아무 서비스로 가서 다른 서비스는 신경 쓰지 않고 독립적으로 작업하게 해 주기는커녕, 이 구성에서는 그 서비스가 다른 서비스에 어떤 영향을 미치는지까지 따져야 한다. 그뿐 아니라 여러 서비스를 함께 수정해야 할 수도 있고, 릴리스 중 호환성을 지키려고 서비스 간 배포를 특정 순서로 맞춰야 한다. 이런 개발·배포 복잡성은 마이크로서비스 시스템의 핵심 이점을 무너뜨린다.

팀이 이런 패턴에 빠져들고 있거나 서비스 간 릴리스를 맞추는 일에 불평이 나온다면, 이는 더 자세히 들여다보고 기술 부채를 갚아 독립적으로 배포 가능한 서비스로 되돌릴 때라는 경고 신호다. 그 기술 부채는 대개 계약, API 설계, 그리고 시스템에서 데이터를 다루는 방식에 자리 잡고 있다.

## 읽기 좋은 좋은 코드 쓰기

전문적인 환경에서 코드 한 줄의 주된 독자는 컴퓨터가 아니라, 언젠가 그 코드를 읽고 이어서 개발해야 할 개발자다. 이것이 프로그래밍의 황금률이다. 엔지니어는 남의 코드에 기대하는 것과 똑같은 수준의 가독성으로 자기 코드를 써야 한다.

### 언어와 생태계의 선택

프로그래밍의 황금률에 따르면, 언어는 팀이 가독성 높고 유지보수하기 좋은 코드를 쓸 수 있게 해 주는 것으로 골라야 한다. 좋은 엔지니어는 보통 어떤 언어로든 그렇게 할 수 있지만, 어떤 언어는 그것을 일관되게 해내기를 다른 언어보다 더 쉽게 만들어 준다. 언어나 생태계를 고를 때 따져 볼 점이 더 있다.

* 그 언어에 익숙한 인재 풀이 얼마나 크고, 더 구체적으로는 그 환경에 익숙하면서 당신 같은 스타트업에도 관심 있는 사람이 얼마나 되는가?
* 출발점으로 삼을 만한 기존 구현이 있는가?
* 특별한 성능·확장 요구사항이 있는가? 특정 종류의 작업에서는 어떤 언어가 다른 언어보다 훨씬 빠르다. 하스켈(Haskell)은 문자열 처리에 비효율적인 것으로 유명하고, C는 대부분의 작업에서 빠른 것으로 유명하다. 물론 특정 문제에서는 C에 근접하거나 그것을 능가하는 속도를 내면서도 더 쉽고 친절한 코딩 환경을 제공하는 언어도 있다.
* 특정 언어에 출발점으로 삼기 좋은 프레임워크가 있는가? 예를 들어 React Native는 강력한 크로스플랫폼 모바일 언어이지만 자바스크립트나 타입스크립트를 요구한다.

엔터프라이즈 환경이라면 Golang, 타입스크립트, Rust처럼 정적 타입 시스템(static type system)을 갖춘 언어를 권한다. 정적 타입 시스템이 있으면 컴파일러가 코드 정확성을 보장하는 궂은일을 더 많이 떠맡아 준다. 코드가 실행되기 전에 도구가 오류를 잡아내는 로컬 개발 환경을 지향해야 한다. 컴파일 시점의 문제를 고치는 것이 런타임 문제를 고치는 것보다 대체로 훨씬 빠르고 비용도 덜 든다.

### 코드 스타일과 포매팅

널리 쓰이는 언어라면, 코드를 어떻게 포매팅해야 하는지에 관한 공개 표준(예: 파이썬의 PEP8)이 있거나, 특정 코드 스타일과 포매팅을 강제할 수 있는 설정형 도구(예: 자바스크립트의 ESLint나 Prettier, C#의 ReSharper)가 있다. 이런 도구는 대부분 누가 작성했든 코드를 스타일 면에서 똑같이 맞추는 일을 아주 잘해 낸다. 코드베이스를 읽기 좋게 만든다는 취지에서, 이런 도구를 쓰지 않고 코드베이스 100퍼센트를 동일한 규칙으로 포매팅하지 않을 변명거리는 없다. 어떤 규칙을 쓸지는 전적으로 당신과 팀의 선호에 달렸지만, 일관되고 읽기 좋은 결과를 내기만 하면 된다.

개발자들이 쓰는 통합 개발 환경(IDE)에서 파일을 저장할 때 코드를 자동 포매팅하도록 설정값이나 안내를 마련해 두기를 권한다. 그런 다음 지속적 통합(CI) 시스템에서 모든 새 코드가 올바르게 포매팅됐는지 확인하면 된다.

주의할 점이 있다. 자동 포매팅 없이 CI 시스템에서 스타일만 강제하면 엔지니어에게 무척 짜증스러운 일이 된다. 그러니 첫날에 모두가 IDE를 올바르게 설정하도록 교육해, CI의 린트 실패로 매번 놀라고 시간을 허비하는 일을 막아야 한다.

### 정적 코드 분석

요즘의 정적 코드 분석(static code analysis)은 보안 허점부터 명백한 버그, 스타일 불일치에 이르기까지 흔한 코드 문제를 폭넓게 식별하고 경고할 수 있다. 이런 도구는 비교적 저렴하고, 널리 쓰이는 다양한 CI 시스템 및 개발자 IDE와 깔끔하게 통합된다. 여러 프로젝트와 프로그래밍 언어에서 이런 도구를 써 본 경험에 비추어 보면, 신호 대 잡음 비가 매우 좋고, 그 결과는 생산성과 소프트웨어 품질 면에서 순이익이다. 소프트웨어 프로젝트 수명에서 비교적 이른 시점에 정적 코드 분석을 도입해야 한다. 자바스크립트용 ESLint처럼 당신이 고른 프로그래밍 언어에 특화된 도구와 더불어, SonarCloud, Codebeat, Scrutinizer-CI, Code Climate, Cloudacity 같은 범용 분석 플랫폼도 함께 살펴보기를 권한다.

### 그린필드 대 브라운필드

그린필드(greenfield) 소프트웨어 개발은 기존 레거시 코드가 거의 없고 도구·패턴·아키텍처를 자유롭게 고를 수 있는 새 환경에서의 개발을 가리킨다. 여기에는 일에 맞는 올바른 아키텍처와 도구를 신중히 고를 수 있고 기존 기술 부채에 발목 잡히지 않는다는 분명한 이점이 있다. 미묘한 단점은, 선택지가 너무 많고 제약이 너무 적은 탓에 잘못된 결정을 내릴 위험이 더 크다는 것이다. 또한 새 프로젝트에는 보통 상당한 부트스트랩 비용이 드는데, 테스트·빌드 시스템·정적 코드 분석 설정처럼 흔히 과소평가되는 것들이 여기에 해당한다.

브라운필드(brownfield) 소프트웨어 개발은 그린필드의 반대로, 기존 레거시 시스템을 다루는 일을 가리킨다. 트레이드오프는 본질적으로 뒤집힌다. 좋든 싫든, 앞선 사람들이 내린 상위 수준의 결정에 묶이게 된다.

브라운필드 개발의 가장 큰 위험은 NIH 증후군(not invented here syndrome)이다. NIH란 자기가 직접 만들지 않은 것에 대해서는 책임지기를 꺼리거나 충분히 주의를 기울이지 않으려는 경향을 말한다. 브라운필드 소프트웨어 개발에서 이는 기존 작업을 이해하는 데 체계적으로 투자가 부족해지는 결과로 이어지고, 기존 시스템을 보강하거나 수정하는 일에 좌절과 비효율을 부른다. 나는 관리자에게, 팀에 기존 시스템을 어떤 식으로든 수정하라고 요구하기 전에 그 시스템을 읽고 이해할 시간을 명시적으로 마련해 주기를 강력히 권한다. 앞단에서 이해에 쓴 시간은 훗날 더 적은 돌발 상황과 더 빠른 속도로 보답받는다.

## 코딩 패턴

어떤 스타일의 코드를 써야 하는가는 많은 개발자에게 종교적인 논쟁거리다. 이 장에서 내 의도는 코딩 패턴에서 가장 흔히 쓰이는 표현들이 무슨 뜻인지 간략히 설명하고, 각 관행을 더 폭넓게 다루는 자료로 안내하는 것이다.

이 주제로 감정적이라고 느껴지는 대화를 마주하게 된다면, 이 패턴들을 하나씩 쓰면서도 성공한 회사가 많다는 점을 기억하라. 모든 것은 트레이드오프다. 형편없는 프로그래머는 어떤 도구로도 엉망을 만들 수 있고, 반대로 뛰어난 프로그래머는 변변찮은 도구로도 읽기 좋은 해법을 찾아낸다.

### 객체지향 프로그래밍(OOP)

객체지향 프로그래밍(object-oriented programming, OOP)은 현실 세계의 명사와 동사를 본떠 코드를 설계하는 방법론이다. 전형적인 예로, 두 사람 사이의 상호작용을 두 개의 Person 객체로 모델링하고, 말하기처럼 사람이 하는 행동은 그 객체의 함수로 두는 것을 들 수 있다. 파이썬, 루비, C#처럼 본질적으로 객체지향인 언어가 많다. 자바스크립트나 C++처럼 객체 선택형(Object-Optional, 객체지향과 함수형 스타일을 어느 정도 모두 지원)인 언어도 있고, 아예 다른 부류인 언어도 있다.

### 순수성

순수한(pure) 코드는 외부 의존성이나 부수 효과가 없다. 달리 말하면, 같은 입력이 주어지면 순수한 코드는 항상 같은 출력을 낸다. 순수한 코드의 이점은 테스트하기 쉽고 외부 설정이나 목(mock)이 필요 없다는 것이다. 또한 순수한 코드는 무슨 일을 하는지 이해하려고 다른 코드를 더 읽을 필요가 없으므로 읽고 이해하기도 쉽다. 순수한 코드의 간단한 예는 두 수를 더하는 함수다. 어떤 두 입력값이 주어지든 합 함수는 항상 같은 출력을 낸다.

본질적으로 순수하지 않은 코드도 있다. 예를 들어 파일시스템, 네트워크, 데이터베이스처럼 바깥세상과 상호작용하는 코드가 그렇다. 그 밖의 대부분은 비즈니스 로직을 순수한 방식으로 모델링할 수 있다. 가능한 한 당신과 팀이 순수한 코드를 쓰기를 권한다.

### 함수형 프로그래밍

코딩 패턴을 품사에 빗대 설명하던 방식을 이어 가자면, 함수형 프로그래밍(functional programming)은 동사(함수)를 시스템의 일급 구성 요소로 다룬다. 대부분의 함수형 프로그래밍은 아주 작은 기능 조각에서 출발해, 그것들을 합성(compose)해 더 정교하고 복잡한 시스템을 만든다. 잘 해냈을 때 함수형 코드의 이점은, 더 순수해지는 경향이 있어 읽고 따져 보고 고립된 채로 테스트하기가 더 쉽다는 것이다. 학술적인 함수형 코드 중에는 형식적으로 따져 볼 수 있는 것, 즉 코드가 올바르게 동작한다는 수학적 증명을 만들어 낼 수 있는 것도 있다.

함수형 프로그래밍을 잘못하면, 매우 장황하고 읽기 어려운 코드가 나올 수 있다. 예를 들어 여러 함수를 합성할 때는, 함수가 몇 개나 합성되는지, 그리고 합성 체인에서 각 함수의 동작이 얼마나 분명한지를 따져 보는 것이 중요하다.

최악의 시나리오를 그려 보자. 함수 열 개가 줄줄이 이어진 체인인데, 각 함수의 이름이 당신에게 아무 의미도 없는 경우다(예: `a(b(c(d(e(f(g(h(i(j(input)))))))))`). 이보다 더 나쁜 경우는, 이 알파벳 함수들의 정의가 코드베이스의 서로 다른 곳에 있는 열 개의 파일에 흩어져 있거나, 더 나쁘게는 서로 다른 임포트 라이브러리에서 온 경우뿐이다.

### 익스트림 프로그래밍과 테스트 주도 개발(TDD)

익스트림 프로그래밍(extreme programming)은 애자일이나 스크럼(SCRUM)과 비슷한 개발 방법론이다. 켄트 벡(Kent Beck)의 책 *Extreme Programming Explained*에 기술된 정식 방법론을 가리킬 수도 있고, 더 비공식적으로는 그 방법론이 옹호하는 코딩 관행 일부를 가리킬 수도 있다. 비공식적인 용법은 그 방법론의 테스트 관행, 구체적으로는 테스트 주도 개발이라는 개념을 가리킨다.

테스트 주도 개발(test-driven development, TDD)은 기능 코드를 먼저 쓰고 테스트를 나중에 쓰는 대신, 기능 소프트웨어보다 테스트를 먼저 쓰는 방식이다. 행위 주도 개발(behavior-driven development, BDD)과 인수 테스트 주도 개발(acceptance-test driven development, ATDD)도 비슷한 관행이다.

### 의존성 주입

의존성 주입(dependency injection)은 특정 객체·모듈·코드 블록의 서비스 의존성을 직접 인스턴스화하지 않고 바깥에서 넘겨받는 패턴이다. 예를 들어 데이터 객체는 설정 파일에서 연결 문자열을 찾아 데이터베이스 클라이언트를 만들어, 자기 데이터베이스 연결을 스스로 인스턴스화할 수 있다. 다른 방법으로는, 부모 격인 호출 코드 블록이 데이터베이스 서비스를 만든 다음, 그 하나의 데이터베이스 서비스를 각 데이터 객체 인스턴스에 넘겨주는 것이다.

의존성 주입의 가장 큰 이점은 서비스와 그 의존성 사이의 결합도를 낮춰, 사실상 둘 사이에 문서화된 인터페이스를 더한다는 것이다. 이 인터페이스 덕분에 예컨대 테스트 맥락에서 목 서비스 같은 다른 구현체를 끼워 쓸 수 있다.

의존성 주입을 깔끔하게 해내는 데는 미묘한 부분들이 있다. 당신이 쓰는 프로그래밍 언어에서 널리 쓰이고 잘 다듬어진 프레임워크나 패턴을 채택하기를 권한다.

### 도메인 주도 설계

도메인 주도 설계(Domain-Driven Design)라는 용어는 에릭 에반스(Eric Evans)가 2003년에 펴낸 책 *Domain-Driven Design*에서 비롯했다. 핵심 아이디어는 비즈니스 도메인의 명사를 모델링하는 모델을 만드는 것이다. 객체지향 설계의 객체를 위한 것이든, 데이터베이스의 스키마를 위한 것이든 말이다. 단순하고 직관적으로 들릴 수 있지만, 복잡한 비즈니스 도메인에서는 코드가 도메인을 일관성 없게 모델링하거나, 팀의 이해를 가로막는 방식으로 모델링하기 십상이다. 특히 더 크고 복잡한 문제에서는, 팀이 자리를 잡고 앉아 문제를 모델링하는 일관된 방식에 합의하고, 시스템 전체에서 같은 개념을 같은 용어로 일관되게 부르도록 늘 강하게 요구한다.

## API 계약

애플리케이션 프로그래밍 인터페이스(application programming interface, API)는 법적 계약과 다르지 않다.

API는 구현에 앞서 설계하고 다듬고 합의하며, 양측은 원하는 결과를 얻기 위해 상대가 계약을 지키리라고 기대한다. API를 설계하고 구현할 때, 당신은 그 API가 특정 방식으로 동작하리라는 약속을 API 소비자에게 하는 셈이다. 법적 계약과 마찬가지로, API가 어떻게 동작할지에 대한 구체적인 생각을 갖고 있더라도 그 미묘한 부분을 상대가 다르게 해석하면 목적을 이루지 못할 수 있다. API의 세부 사항은 정말로 중요하며, 팀이 API를 일관되고 효율적인 방식으로 설계·구축하도록 보장하는 것이 기술 리더로서 당신의 역할이다.

그렇지만 고품질 API를 만드는 일은 의외로 복잡한 작업이다. 인터페이스 설계, 로직·데이터를 처리하는 코드 구현, 기능 테스트, 문서 작성, 버전 관리·변경 관리, API가 바뀔 때 문서를 최신으로 유지하는 일, 그리고 개발자가 API와 쉽게 상호작용하도록 만드는 일까지 고려할 것이 많다. 이것들을 잘 해내느냐 못하느냐가, 개발자가 사랑하는 API와 구현을 가로막고 중요한 프로젝트의 출시 시간을 늦추는 API의 차이를 가른다. 리더로서 이런 일들을 잘 다루기 위해 쓸 수 있는 주요 레버는 두 가지, 거버넌스와 아키텍처다.

### API 설계 거버넌스

API를 구축하는 모든 요소에는 수많은 결정이 들어간다. 좋은 API와 나쁜 API를 가르는 것은 그 결정들의 일관성, 예측 가능성, 정확성이다. 기술 리더로서 당신에게는 조직 전반에 걸쳐, 개발자들이 서로 일관되고(서로 어긋나지 않고), 예측 가능하며(당면한 문제에 적합한 공통 패턴을 쓰고), 정확한 API를 만들도록 돕는 구조를 갖추는 일이 주어진다.

이 목표를 이루려면 어떤 형태로든 거버넌스 체계가 필요하다. 명확히 문서화된 지침과 표준에서부터, 모든 API를 정기적으로 검토하고 승인하는 책임을 진 사람들의 그룹에 이르기까지 다양하다. 팀이 클수록 높은 기준을 유지하기 위해 프로세스와 거버넌스에 더 많은 시간과 노력을 들여야 한다.

### API 아키텍처

실무에서 마주치게 될 API는 크게 HTTP 기반과 비(非)HTTP 기반 두 가지다. 다른 모든 도구와 마찬가지로 HTTP에도 트레이드오프가 있어 모든 일에 이상적이지는 않다. 그래서 비즈니스 요구사항이 초저지연이나 초고처리량·저오버헤드, 또는 실시간 스트리밍 애플리케이션을 요구한다면 HTTP를 넘어서는 무언가를 찾게 될 것이다. 아래에서는 HTTP API 몇 가지를 다룬 다음, 마주칠 법한 비HTTP API도 간략히 살펴본다.

### HTTP 기반 API

웹이나 모바일 애플리케이션, 심지어 대부분의 시스템 백엔드를 만들고 있다면, 주로 다루게 될 것은 십중팔구 HTTP API다.

*XML과 SOAP API*

2000년대 초에는 가장 흔한 API 패턴이 XML 기반의 단순 객체 접근 프로토콜(Simple Object Access Protocol, SOAP)이었다. SOAP를 비롯한 XML 기반 API 스타일은 2020년대 스타트업 사이에서는 완전히 한물갔지만, 레거시 시스템에는 여전히 흔하다. 특히 기술 변화가 더딘 산업의 대기업에서 그렇다. 새 SOAP나 XML 기반 API를 만들어서는 안 된다.

*REST*

REST(Representational State Transfer)는 JSON을 HTTP 위에 얹어 API로 쓰는 방식을 가리키는 포괄적인 표현이다. REST는 때로 HATEOAS라는 패턴으로 보강되는데, 이는 REST API의 콘텐츠·페이로드에 더 형식적인 표준 집합을 제공한다. HATEOAS(그리 흔하지는 않다)가 없으면, REST에는 JSON 데이터를 어떻게 모델링할지에 대한 형식적이거나 브랜드화된 지침이 없다. REST API는 보통 하나의 명사를 하나의 엔드포인트로 모델링하고, HTTP 동사(GET, PUT, POST, DELETE 등)로 그 명사에 대한 동작을 결정한다. 예를 들어 GET `/users`는 사용자 목록을 가져오고, POST `/users`는 새 사용자를 만들며, DELETE `/users/123`은 ID가 123인 사용자를 삭제한다.

REST는 당신이 마주칠 API 형태 중 십중팔구 가장 흔하다. REST는 도구 생태계가 넓고 든든하며, 거의 모든 엔지니어가 익숙하다.

*GraphQL*

GraphQL은 JSON을 HTTP 위에 얹는다는 점에서 REST와 비슷하지만, HTTP 동사에 기대지 않는다. GraphQL에서는 거의 모든 것이 POST이며, 쿼리(query)와 뮤테이션(mutation)으로 이뤄진 구조화된 스키마를 쓴다.

나는 GraphQL을 타입과 자기 문서화 스키마를 갖춘 REST라고 생각하기를 좋아한다. 그래서 GraphQL API에는 자동 생성되는 문서와 스키마 탐색기가 따라오는 경향이 있다. 또한 GraphQL은 스키마 시스템 덕분에 여러 서비스의 여러 스키마를 합성해 더 크고 강력하며 복잡한 데이터 그래프를 만들 수 있는데, 이를 페더레이티드 스키마(federated schema)라고 부르기도 한다. 아폴로(Apollo)라는 회사가 그래프를 관리하고 확장하는 정교한 솔루션을 제공한다.

회사의 데이터를 모델링하는 그래프를 만드는 일의 이점, 그리고 스키마를 설계하도록 강제됨으로써 길러지는 좋은 습관에 대해서는 할 말이 많다. 그렇지만 트레이드오프 없는 시스템은 없다. GraphQL은 표준 HTTP 동사를 쓰지 않기 때문에 웹 스택의 일부 요소와 잘 어울리지 못한다. GET 요청 캐싱과 개발자 도구는 GraphQL 요청을 매끄럽게 다루는 데 여전히 따라잡는 중이다. 이런 단점이 당신 비즈니스에 큰 걱정거리가 아니라면, apollographql.com을 꼭 살펴보고 GraphQL 사용을 고려해 보기를 강력히 권한다. 특히 사내용 API라면 더욱 그렇다.

### 비HTTP API

일반적으로 전통적인 동기식 요청/응답(즉 원격 프로시저 호출, RPC) 방식의 API라면, 어디에나 통하는 보편성 덕분에 HTTP API를 쓰는 편이 좋다. 다만 특히 비동기 작업에서는 HTTP에 깔끔하게 들어맞지 않는 API 패턴이 몇 가지 있고, 흔히 쓰이는 대안 구현들이 있다.

*큐잉 시스템*

큐잉 시스템(queueing system)은 메시지를 받는 인박스(또는 인박스 집합)와, 소비자가 일정한 보장 아래 메시지를 읽을 수 있는 인터페이스를 유지한다.

전형적인 큐잉 시스템은 메시지 순서(선입선출 FIFO 또는 후입선출 LIFO)와 더불어, 최소 한 번(at least once) 또는 최대 한 번(at most once) 전달을 보장할 수 있다. 대부분의 클라우드 플랫폼에는 AWS Simple Queue Service(SQS)나 Google Cloud Task 큐처럼 호스팅된 큐 구현이 있다.

큐잉 시스템에는 흔히 *명시적(explicit)* 호출이라는 개념이 있다. 발행자(publisher)가 메시지를 만들 때 그 요청을 어떻게 처리하거나 실행할지를 명시적으로 지정한다는 뜻이다. 이와 대조적으로 대부분의 발행-구독 시스템은 *암묵적(implicit)* 실행을 지원한다. 즉 발행자는 어떤 시스템이 그 메시지를 처리할지 미리 알 필요 없이, pub/sub 시스템이 메시지를 전달하리라는 것만 알면 된다.

*발행-구독(pub/sub) 패턴*

발행-구독 패턴(publisher-subscriber pattern), 줄여서 pub/sub은 여러 출처에서 메시지가 만들어지고 다양한 패턴으로 여러 구독자에게 전달되는 시스템을 설계할 수 있게 해 준다. 발행자-구독자 관계는 일대일(직접), 일대다(팬아웃, fan-out), 다대일(팬인, fan-in), 다대다로 모델링된다. 여러 pub/sub 구현은 메시지가 모든 구독자에게 전달된다거나, 적어도 한 구독자에게 전달된다거나, 적어도 한 번 전달된다는 식의 보장을 제공할 수 있다. 큐와 마찬가지로 RabbitMQ 같은 기성 솔루션도 있고, Amazon Simple Notification Service(SNS)나 Google Cloud Pub/Sub처럼 손쉽게 확장되는 클라우드 호스팅 옵션도 있다.

pub/sub 패턴과 그것이 제공하는 보장은 대단히 강력하다. 다만 트레이드오프는, 광고된 보장을 실현하려면 구현에 상당한 주의와 세심함이 필요하다는 것이다. 예컨대 구독자를 구현하려면 메시지 확인(acknowledgement) 의미 체계에 세심한 주의를 기울이고, 올바른 메시지가 올바른 곳으로 가도록 토픽 구독을 신중하게 관리해야 한다.

큐, pub/sub, HTTP API 중 무엇으로 구현할지 망설여진다면, 내 일반적인 권고는 단순하게 가서 동기식 HTTP API를 택하라는 것이다. 구현 방식 사이에서 망설이고 있다는 사실 자체가, 비동기 시스템이 제공하는 보장이 당신 구현에 결정적이지 않다는 뜻이고, 따라서 늘어나는 복잡성은 당신의 스타트업 프로젝트에 들일 가치가 없을 가능성이 크다.

*잡 시스템*

잡(job), 즉 크론 잡(cron job)은 발행자나 클라이언트가 아니라 어떤 형태의 타이머가 거의 대부분 트리거하는 백엔드 API의 한 종류다. 흔한 예로 야간 데이터 정리 작업이나 주간 이메일 요약·알림 발송이 있다. 잡에 관한 몇 가지 모범 사례:

* 직접 만들지 말고, 남이 유지보수하는 잡 시스템을 써라.
* 잡 시스템을 고를 때(정 만들어야겠다면 만들 때도) 다음을 갖췄는지 확인하라.
  - 모든 잡 실행에 대한 로깅이 있다.
  - 실패한 잡의 재시도를 설정할 수 있다.
  - 잡이 실패하면 알림을 준다. 엔지니어가 예약 잡을 설정하고 첫날에는 잘 도는 것을 지켜보다가, 15일째 잡이 실패했는데 아무도 알아채지 못하고 30일째에야 발견하는 일은 아주 흔하다.
  - 잡과 잡 상태를 볼 수 있는 인터페이스를 제공한다.
  - 잡 설정을 코드나 구성으로 소스 관리에 저장할 수 있다.
  - 다른 내부 시스템 API·자원에 접근하기 위해, 필요에 따라 잡을 당신의 환경·사설 네트워크·보안 그룹 안에서 실행할 수 있다.
  - 비밀 관리(secret management) 시스템과 통합된다.
  - 개발 환경과 프로덕션 환경에서 잡을 손쉽게 설정하고, 각 환경에서 쉽게 테스트할 수 있다.

## 문서화

API에 대해 빠짐없고 명확하며 최신인 문서를 갖추는 일은, API를 어떻게 구축하고 유지보수하느냐만큼이나 중요하다. 훌륭한 API 문서의 몇 가지 핵심 특징:

* 항상 구현과 일치한다
* 가능한 모든 입력과 그 타입을 문서화한다. 가능한 모든 오류를 문서화한다
* 다른 엔지니어가 읽고 탐색하기 쉽다

API를 만들 때는 API 문서 생성 기능이 포함된 시스템으로 만드는 것이 언제나 좋다. 그렇게 하지 않으면 위의 목표들을 꾸준히 모두 충족하기란 사실상 불가능하다. REST API를 만든다면, OpenAPI(당신의 API를 기술하는 YAML 또는 JSON 문서)로 API를 설계하기를 강력히 권한다. 대부분의 언어에는 OpenAPI 스펙을 받아들여 스펙에 맞는 컨트롤러·라우트를 자동 생성하거나, 구현된 API가 스펙과 일치하는지 확인하는 테스트 하니스를 생성해 주는 SDK가 있다. 그뿐 아니라 stoplight.io나 readme.com 같은 온라인 도구는 OpenAPI 문서를 받아 보기 좋고 탐색하기 쉬운 문서를 생성해 준다.

GraphQL을 쓴다면, GraphQL Playground나 Apollo Studio 탐색기가 방대한 타입 문서를 어느 정도 대신해 줄 수 있다. 그래도 나는 readme.com 같은 도구를 쓰든 손수 만들든, 입문서나 시작 가이드 역할을 할 별도의 API 문서 페이지를 만들기를 권한다. GraphQL에 내장된 문서에는 인증이 어떻게 동작하는지에 대한 설명이 없고, API 안의 데이터 사이의 관계를 설명할 공간을 마련하는 일도 서툴다.

이런 빈틈은 다른 데서 메워야 한다.

OpenAPI나 GraphQL을 쓰는 또 다른 이점은, 그 결과로 나오는 API 명세가 문서 생성기와 테스트 프레임워크뿐 아니라 Insomnia나 Postman 같은 개발자 IDE로도 이식된다는 것이다. 이런 IDE 덕분에 개발자는 코드를 짜지 않고도 API와 빠르게 상호작용하며 기능을 검증할 수 있다. 형식 명세는 코드 생성 도구와 함께 써서 코드의 타입 일관성을 보장하는 데도 쓸 수 있다.

## 멱등성

API 요청이 멱등(idempotent)하다는 것은, 같은 요청을 여러 번 보내도 한 번 보낸 것과 같은 효과를 낸다는 뜻이다. 멱등성(idempotency)은 견고한 시스템을 구축하고 데이터 손상을 피하는 데 중요한 개념이다. 다른 모든 것이 그렇듯, 멱등성은 시스템에 유용한 보장을 주지만 대가가 따른다. 멱등성을 구현하면 백엔드 시스템의 복잡성이 늘어난다.

REST API에서는 POST를 제외한 모든 HTTP 동사가 멱등해야 한다고 널리 가정한다. 예를 들어 GET 요청은 정의상 같은 입력에 대해 항상 같은 결과를 반환해야 한다(물론 기반 데이터가 바뀌지 않는 한). 일반적으로 PUT 요청은 기존 객체를 수정하므로 자연스럽게 멱등해야 한다. 반면 POST 요청을 여러 번 호출하는 것은, 대부분의 시스템에서 여러 객체를 만들겠다는 의도를 나타낸다.

### 멱등성 키

REST의 HTTP POST 요청과 GraphQL 뮤테이션 API에서는 멱등성이 표준·명세 차원에서 제공되지 않는다. 클라이언트가 이런 종류의 요청을 재시도하면서도 멱등하게 동작하게 하고 싶다면, 멱등성 키(idempotency key) 패턴을 구현해야 한다. 멱등성 키는 클라이언트가 제공하는 임의의 문자열로(HTTP 헤더로 주거나, GraphQL이라면 입력 변수로 줄 수 있다), 백엔드는 이를 이용해 들어오는 요청의 중복을 제거한다. 이를 위해 백엔드는 멱등성 키를 저장해야 하고, 그 키로 들어온 요청의 응답도 저장해 두었다가 나중에 클라이언트에게 돌려줘야 한다.

멱등성 키를 구현하는 일은 만만치 않다는 점에 유의하라. 추가적인 데이터베이스 쓰기, 요청 응답을 갈무리하는 로직, 그리고 동시에 도착한 중복 요청에 대한 동시성·잠금 문제를 다루는 일이 필요하기 때문이다. 만약 당신 애플리케이션에서 멱등성이 중요하다면 예컨대 금융 거래를 다룬다면 그것을 처음부터 직접 만들기보다, 견고한 멱등성 시스템을 기본으로 제공하는 백엔드 API 구현을 채택하기를 권한다.
