# 데이터와 분석

대부분의 스타트업은 사업을 운영하면서 적어도 세 종류의 데이터를 사용한다.

* 트랜잭션 데이터

* 분석용 비즈니스 인텔리전스 데이터

* 행동 데이터

세 데이터는 각각 양이 다르고, 읽기/쓰기 패턴도 다르며, 시각화하고 인사이트를 끌어내는 데 필요한 도구도 다르다.

빅데이터라는 표현을 잠깐 짚고 넘어가자. 스타트업이라면 십중팔구 무한 확장(또는 웹 스케일)을 염두에 두고 설계해야 할 만큼의 빅데이터를 다루고 있지는 *않을* 것이다. 적당한 양의 하드웨어에 그럭저럭 괜찮은 데이터 모델만 얹은 평범한 기성 데이터베이스로도 수천만 행, 수백 기가바이트의 데이터를 받아들일 만한 성능으로 충분히 처리한다. 데이터 파이프라인이나 데이터 웨어하우스 어플라이언스 같은 대다수 빅데이터 솔루션은 설치와 구성이 훨씬 복잡하고, 지연 시간과 비용도 만만치 않게 늘어난다. 당신의 스타트업에는 과한 선택일 가능성이 크다. 단순하게 가자. 일반적인 데이터베이스(예: PostgreSQL)로는 도저히 감당이 안 된다는 설득력 있는 근거를 댈 수 있을 때에만 빅데이터 솔루션을 고려하라. 달리 말하면, 데이터베이스 아키텍처를 섣불리 최적화하지 마라.

### 트랜잭션 데이터

트랜잭션 데이터는 애플리케이션 자체를 굴리는 데이터로, 보통 주력 NoSQL 또는 SQL 데이터베이스에 들어 있다. 트랜잭션 데이터는 매우 낮은 레이턴시와 높은 가용성을 요구하지만, 전체 크기는 다른 형태의 데이터에 비하면 크지 않다. 추천하는 방법은 기성 SQL이나 NoSQL 솔루션, 그중에서도 MongoDB Atlas나 Google Cloud SQL처럼 호스팅까지 대신 해 주는 서비스를 고르는 것이다. 프로덕션 데이터베이스에 있으면 좋은 기능 몇 가지를 꼽자면 다음과 같다.

* 원클릭 특정 시점 복구
* 정기 백업과 원클릭 복원
* 부하 경감(load shedding)을 위한 읽기 전용 복제본
* 가용성을 위한 멀티 존 복제·호스팅
* 이벤트 기반 감사 로깅
* 디스크 자동 확장/축소
* 연결/IP 수준 보안
* 리소스(CPU, RAM, 디스크, 네트워크) 모니터링·알림, CPU/RAM 원클릭 스케일 업/다운
* 느린 쿼리 모니터링

### 분석용 비즈니스 인텔리전스 데이터

비즈니스 인텔리전스(BI)는 사용자의 행동을 들여다보는 데 쓰는 데이터로, 대개 트랜잭션 데이터에서 뽑아낸다. 초기에는 BI 쿼리를 트랜잭션 데이터베이스에 대고 직접 돌려도 대체로 별 탈이 없다. 그러나 데이터 크기와 쿼리 복잡도가 커질수록, 높은 가용성과 낮은 레이턴시가 필요한 시스템에 부하를 더하는 셈이라 점점 골치 아파진다. 이때 자연스러운 해법은 트랜잭션 데이터베이스의 읽기 전용 복제본에 쿼리를 돌리거나, 데이터 파이프라인을 통해 데이터를 다른 저장 시스템으로 복사·변환하는 것이다.

데이터 파이프라인 구축과 데이터 웨어하우징은 그 자체로 책 한 권 분량이고, 최신 기술도 끊임없이 바뀐다. 큰 틀에서 몇 가지만 조언하겠다.

주력 BI 데이터 웨어하우스로는 Snowflake, Databricks, Google BigQuery 같은 엔터프라이즈 데이터 솔루션을 살펴보라. 이런 도구는 판도를 바꿔 놓는다. 특히 서버리스 웨어하우스(BigQuery, Aurora)는 설정이 식은 죽 먹기이고, 데이터 크기와 상관없이 레이턴시가 꽤 일정하며, 초·중기 스타트업에는 비용 효율도 매우 높다.

요즘은 스타트업이 정교한 데이터 파이프라인 아키텍처를 직접 구축하고 운영할 필요가 없다. ELT(Extract, Load, Transform)와 ETL(Extract, Transform, Load) 도구를 이제 엔터프라이즈 데이터베이스의 데이터 레이크/웨어하우스 안에서 통째로 돌릴 수 있고, dbt 같은 도구는 재현성과 테스트 가능성, 그리고 파이프라인을 코드로 다루는 기능까지 제공해 데이터 파이프라인 운영을 한결 수월하게 만든다.

데이터 시각화에는 Looker, Domo, Preset 같은 호스팅형 또는 클라우드 네이티브 솔루션을 고려해 보라.

엔지니어링 팀과 제품 팀이 데이터와 비즈니스 인텔리전스를 책임지는 담당자와 긴밀히 협업하도록 챙겨라. 제품 기획 과정의 초반부터 데이터 관점을 끌어들이면 "두 번 재고 데이터 스키마는 한 번에 만든다"는 자세로 훗날의 골칫거리를 크게 덜 수 있다.

### 행동 데이터

행동 데이터(behavioral analytics events라고도 한다)는 사용자가 애플리케이션을 어떻게 사용했는지를 담는다. 행동 데이터는 대체로 양이 꽤 많지만 스키마는 비교적 단순하고, 강력한 시각화 소프트웨어와 함께 쓸 때 가장 빛을 발한다.

대체로 애플리케이션에서 나온 행동 데이터는 여러 곳으로 보내고 싶을 것이다. 여기서 일종의 라우팅 문제가 생긴다. 데이터 출처는 하나(애플리케이션)인데 그 이벤트를 여러 곳으로 보내야 하기 때문이다. 이 문제의 거의 표준이 된 해법은 Twilio의 Segment 플랫폼이며, RudderStack처럼 고객 데이터 플랫폼(CDP, Customer Data Platform)이라 불리는 떠오르는 대안도 있다. CDP는 애플리케이션에서 데이터를 받아들인 뒤, 이를 데이터 웨어하우스로, 그리고 원하는 만큼 많은 다른 SaaS 플랫폼으로 보내 준다.

애플리케이션이 만들어 내는 행동 데이터가 트랜잭션 데이터와 크게 다른 점 하나는 정확성이다. 행동 데이터는 대부분 유실이 있다. 사용자가 광고 차단기를 쓰거나, 요청이 누락되거나, 방화벽이 가로막는 식이다. 이벤트가 클라이언트 기기에서 CDP까지 도달하지 못하는 이유는 많다. 그렇다고 행동 데이터가 쓸모없다는 뜻은 아니지만, 유실이 있다는 사실을 염두에 두고 데이터에 거는 기대치를 잡고 쿼리할 때의 활용 범위를 한정해야 한다. 정확한 수치가 필요하다면 BI 플랫폼과 트랜잭션 데이터에서 끌어내야 한다고 보면 된다.

### 아키텍처 설계를 위한 일반 팁과 모범 사례

아키텍처 설계에 관한 전반적인 권고 몇 가지로 이 절을 마무리하자.

#### 비즈니스 로직은 백엔드에 두라

애플리케이션을 만들다 보면 로직을 어디에 둘지 선택해야 하는 순간이 자주 온다. 클라이언트(예: 웹 브라우저, 모바일 기기, 물리적 하드웨어)냐, 아니면 어떤 형태의 백엔드 서버냐다. 인증과 관련된 것, 금액 계산, 부정 행위·변조 방지 메커니즘처럼 특정 종류의 로직은 백엔드에 두는 것이 확고한 요건이다. 나머지 대부분의 로직도 다음과 같은 이유로 백엔드에 두는 편이 좋다.

백엔드는 클라이언트보다 테스트하기 쉬운 경우가 많아, 비즈니스 로직의 정확성을 더 자신 있게 확인할 수 있다.

백엔드에 로직을 많이 둘수록 클라이언트는 가벼워지고, 여러 플랫폼용 클라이언트가 로직이라는 단일 출처를 공유할 수 있어 코드 중복도 줄어든다.

백엔드에 있는 로직은 클라이언트가 변조하거나 바꿀 수 없다.

#### 서비스를 외부에 공개할 수 있게 만들라

백엔드에서 다른 백엔드로, 또는 백엔드에서 프론트엔드로 이어지는 API는 제3자가 가져다 쓸 수도 있는 범용 API라고 생각해야 한다. 그래야 몇 가지 좋은 설계 습관을 지키게 된다. 인터페이스가 그 자체로 이해되도록 만드는 일(도메인 주도 설계), 합리적인 인증 메커니즘을 쓰는 일, 데이터 설계에서 적절한 상위 수준의 소유권 추상화를 두는 일이 그렇다. 게다가 언젠가 어떤 서비스를 외부에 공개하고 싶어지는 날이 와도, 거기까지 가는 길이 훨씬 짧아진다.

#### 언어는 되도록 적게 쓰라

프로그래밍 언어마다 그에 딸린 빌드 시스템, 의존성 관리 시스템, 프로그래밍 모범 사례, 인터페이스가 따라온다. 팀은 주력 언어와 그 생태계가 로컬 개발자, 테스트 환경, 프로덕션 환경 모두에서 잘 통합되고 제대로 돌아가도록 상당한 공을 들여야 한다.

스택에 언어를 하나 더 추가할 때마다 그 모든 수고를 똑같이 다시 들여야 하고, 런타임끼리 코드를 공유하지 못하는 불편까지 떠안게 된다. 스택에 새 언어를 들이기 전에, 그 언어가 가져다주는 이점이 그것이 더하는 운영·유지보수 부담을 압도한다는 탄탄하고 빈틈없는 논거를 댈 수 있어야 한다. 그러지 못한다면 그 언어는 없는 편이 낫다.
