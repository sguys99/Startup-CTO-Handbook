# 테스트

소프트웨어 코드를 테스트하는 대신, 당신이 시(市)의 다리를 점검하는 검사관이라고 상상해 보자. 다리는 이미 다 지어졌고, 이제 그것을 점검해 일반에 개통해도 좋을지 판단하는 것이 당신의 일이다. 다리에 박힌 볼트와 리벳을 하나도 빠짐없이 다 검사할 것인가? 그러면 다리가 안전하다는 확신은 꽤 높아지겠지만 시간도 한참 걸려서, 다음 주에 다리를 개통하려던 시장의 계획은 어그러진다.

또 하나의 합리적인 전략은 다리가 정해진 안전계수를 충족하는 데 정확히 어떤 요소가 필수인지 가려내어 그 부분만 테스트·점검하는 것이다. 리벳을 하나 걸러 하나씩 검사하고, 케이블과 구조용 콘크리트는 전부 검사하는 정도면 될 수도 있다.

소프트웨어 엔지니어링의 테스트도 마찬가지다. 목표는 커버리지를 위한 커버리지가 아니라, 소프트웨어가 의도한 대로 동작한다는 확신을 주는 커버리지다.

효과적인 소프트웨어 테스트가 항상 100퍼센트 코드 커버리지를 뜻하지는 않는다. 좋은 테스트 스위트의 기준은, 빌드가 초록불이고 모든 테스트가 통과했을 때 소프트웨어를 최종 사용자에게 배포할 준비가 되었다는 확신을 팀에 주느냐다. 그 확신을 주는 데 100퍼센트 커버리지가 필요할 수도 있고, 30퍼센트면 충분할 수도 있다. 정확한 수치는 당신이 직접 정하고 지켜봐야 한다. 시간이 지나 테스트가 예전만큼 확신을 주지 못한다는 걸 알게 되면(반대로 과하게 투자하고 있다는 걸, 즉 테스트 스위트에 자원을 잔뜩 쏟는데도 버그가 너무 자주 새어 나간다는 걸 알게 되면) 그 노력의 양도 달라진다.


## 테스트/품질 보증 팀

팀 규모에 따라 전담 테스트 팀이 아예 없을 수도, 한 종류 혹은 여러 종류의 테스트를 맡는 테스트 팀이 하나 있을 수도, 다양한 종류의 소프트웨어 테스트를 담당하는 테스트 팀이 여럿 있을 수도 있다. 누가 테스트를 하든, 소프트웨어 테스트가 복잡한 과정이며 미묘한 차이가 중요하다는 점을 인식하는 것이 핵심이다. 소프트웨어를 효과적으로 테스트하려면, 코드를 직접 짠 사람이든 남이 떠넘긴 코드를 받은 사람이든 테스터는 그 코드/소프트웨어가 무엇을 해야 하는지 깊이 이해하고 있어야 한다. 당신의 역할은 팀들이 서로 공감할 수 있도록 환경을 만드는 것이다. 그러려면 팀들이 목표/KPI를 공유하게 하고, 개발자와 테스터 사이에 탄탄하고 끊임없는 소통이 오가는 프로세스를 갖추며, 팀들이 건강하고 생산적인 관계를 유지하는지 지켜봐야 한다.

## 테스트 품질

서로 다른 소프트웨어 테스트 패러다임의 세부 사항으로 들어가기 전에, 소프트웨어 테스트의 목적이 무엇인지, 따라서 무엇이 좋은 테스트를 만들고 반대로 무엇이 나쁜 테스트를 만드는지 한번 생각해 볼 만하다.

나쁜 테스트를 정의하기는 간단하다. 나쁜 테스트는 팀에 주는 이득보다 비용이 더 크다. 나쁜 테스트의 흔한 특징은 다음과 같다.

* 정당한 코드 변경에 맞춰 테스트를 유지·보수하는 데 드는 시간이, 버그를 잡아 아끼는 수고보다 더 크다.

* 테스트의 거짓 양성(false positive) 비율이 높거나 결과가 들쭉날쭉해서, CI를 느리게 만들고 개발자들이 헛된 실패를 다시 돌리느라 짜증과 컨텍스트 스위칭 비용을 치른다.

* 테스트가 엉성하게 설계되어, 코드가 잘못된 동작을 하는 것을 말 그대로 정상으로 검증한다.

* 스위트가 코드의 올바른 동작을 검증하고 거짓 양성도 낮지만, 너무 뒤엉키고 이해하기 어려워서 그 테스트를 짠 사람만 새 테스트를 추가할 수 있고, 다른 엔지니어들은 들여다볼 때마다 두통을 얻는다.

* 테스트가, 평가 대상 코드를 최종 사용자에게 출시해도 된다는 확신을 주지 못한다.

이런 그림을 머릿속에 그려 두면, 대조적으로 좋은 테스트가 어떤 속성을 갖춰야 하는지도 비교적 쉽게 보인다. 좋은 테스트는 다음과 같다.

* 로컬에서든 공유 CI 환경에서든 쉽고 빠르게 돌릴 수 있다

* 안정적으로 실행되어 같은 결과를 일관되게 내며, 보강하기 쉽다

* 바탕 로직이 바뀌었을 때 리팩터링하거나 갱신하기 쉽다

* 바탕 코드 패턴에 적절히 결합되어, (말하자면) 올바른 고도에서 테스트함으로써 의미 있는 고장을 잡아낸다

* 어떤 엔지니어든 이해하고 손볼 수 있다

테스트 접근법을 평가하는 가장 좋은 방법 중 하나는 가장 뻔한 방법이기도 하다. 바로 팀이 테스트를 어떻게 느끼는지 간단한 감정 분석으로 물어보는 것이다. 결과는 대개 매우 이분법적이다. 테스트가 팀이 의지하는 안전망이어서 분명한 순가치를 더해 주기에 자연스럽게 보강하게 되거나, 아니면 테스트가 충분한 가치도 없이 생산성만 갉아먹는 골칫거리라서 팀이 소극적으로, 심지어 적극적으로 싫어하거나 둘 중 하나다.

## 무엇을 테스트할 것인가

팀은 시스템이 올바르게 동작한다는 확신을 높이면서도, 시스템이 유기적으로 커지는 와중에 그 테스트를 장기적으로 유지하는 부담은 최소화하는 방식으로 테스트를 추가해야 한다. 이 고통을 줄이는 일반적인 패턴은 공개 인터페이스(public interface)를 테스트하는 것이다. 공개 인터페이스는 잘 설계되어야 하고 시간이 지나도 비교적 안정적이어야 한다.

또한 공개 인터페이스는 소프트웨어 사용자가 실제로 경험하게 될 정상 경로(happy path)이기도 하다.

따라서 공개 인터페이스에 대한 테스트는 시간이 지나도 거의 바뀌지 않으면서, 중요한 코드 부분이 올바르게 동작한다는 확신을 준다.

소프트웨어의 공개 인터페이스는 프로젝트마다 다르다. 많은 프로젝트에서 그것은 실제 HTTP 기반 API일 것이고, 어떤 프로젝트에서는 라이브러리나 내부 서비스의 함수/클래스 집합일 것이며, 또 어떤 프로젝트에서는 사용자 인터페이스일 수도 있다.

## 테스트 유형 비교

소프트웨어 테스트는 다음 범주/패러다임으로 나눌 수 있다.

* 단위 테스트(unit testing)
* 통합 테스트(integration testing)
* 엔드투엔드 테스트(end-to-end testing)
* 수동 테스트(manual testing)
* 반자동 테스트(semi-automated testing)

이 테스트 유형들을 구분 짓는 속성은 다음과 같다.

**코드 설계(Code planning):** 해당 테스트 패러다임에서 테스트 가능하도록 코드를 짤 때 얼마나 많은 사전 고려가 필요한가.

**테스트 범위(Test scope):** 각 테스트가 한 번에 얼마나 많은 부분을 평가할 수 있는가.

**변경 감지 단위(Change granularity detection):** 어느 크기 또는 어떤 종류의 코드 변경을 테스트가 감지해 실패를 일으킬 가능성이 있는가.

**실행 비용(Run cost):** 테스트를 돌리는 데 시간 또는 비용이 얼마나 빠르게 혹은 많이 드는가.

**추가 노력(Addition effort):** 커버리지를 더 늘리는 데 얼마나 많은 노력이 드는가.

**셋업 노력(Setup effort):** 효과적인 테스트 스위트를 구축하는 데 얼마나 많은 노력이 드는가.

다음 표는 다섯 패러다임이 이 지표들에 어떻게 대응되는지 요약한 것이다. 완벽한 테스트 패러다임은 없으며, 각각 트레이드오프가 있으니 당신의 회사와 코드베이스에 어떤 트레이드오프가 맞는지 신중히 따져 보길 권한다. 정답은 대개 여러 테스트 패러다임을 섞어, 각 유형을 그것이 당신의 애플리케이션에서 가장 큰 가치를 더하는 곳에 쓰는 것이다.

## 단위 테스트

소프트웨어 테스트라고 하면 흔히 가장 먼저 떠오르는 것이 단위 테스트다. 학교에서 두루 가르치고 교과서에도 실리며, 현실에서도 대개 가장 많은 노력이 들어간다. 이 모든 점을 보면 단위 테스트가 최고의 테스트 유형일 것 같지만, 나는 늘 그렇지는 않다고 본다. 우선 단위 테스트를 명확히 정의해 다른 테스트 패러다임과 구분하는 것부터 시작하자.

단위 테스트는 외부 서비스에 대한 네트워크 연결이나 의존 없이, 평가 대상 코드와 같은 메모리 공간에서 한 머신의 메모리 안에서 전부 실행된다. 대부분의 단위 테스트는 매우 빠르게 돌고, 한 번에 아주 적은 양의 코드를 테스트하며, 시작하기도 비교적 쉽다. 또한 단위 테스트는 대개 코드 계약(code contract)에 단단히 결합되어 있고, 평소라면 쓸 수 있는 외부 의존성 없이도 테스트 대상 코드가 실행되도록 모킹(mock) 형태의 새 코드를 요구하는 경우가 많다.

단위 테스트가 치르는 핵심 트레이드오프이자 가장 큰 단점은 테스트 대상 코드에 단단히 결합된다는 점이다. 단위 테스트가 실제 함수 호출과 내부 데이터 객체에 깊이 얽히기란 너무도 쉽다. 이렇게 깊이 의존하면, 코드를 리팩터링할 때 단순하고 무해한 변경이라도 단위 테스트까지 상당히 손봐야 한다. 또한 단위 테스트가 돌아가게 하려고 모킹과 테스트 데이터 픽스처를 만드는 데도 짜고 유지할 코드가 상당히 들어가는 경우가 많아, 단위 테스트 프레임워크에 예상치 못한 비용을 보탠다.

## 통합 테스트

통합 테스트는 단위 테스트의 인메모리·무의존성 제약을 푼다. 그래서 통합 테스트는 더 느리게 도는 경향이 있고, 테스트 대상 코드와 더 높은 수준에서 통합된다. 테스트 대상 코드와 다른 프로세스에서 돌 때는 보통 API 같은 외부로 노출된 계약을 검증한다. 즉 API 동작을 바꾸지 않는 내부 리팩터링은 통합 테스트에 잘 걸리지 않으므로, 이 테스트는 전반적으로 덜 부서지지만 작은 부작용을 감지할 가능성도 낮다.

게다가 통합 테스트는 모킹을 더 적게 만들거나 아예 만들지 않아도 되므로, 구현에 들어가는 코드가 더 적을 수 있다.

## 엔드투엔드 테스트

엔드투엔드(e2e) 테스트는 최종 사용자와 같은 방식으로 코드를 실행한다. 백엔드 코드의 경우, 엔드투엔드 테스트와 통합 테스트는 외부 계약을 통해 코드를 돌린다는 점에서 동일하게 작동할 수 있다. 사용자 대면 프런트엔드 코드라면, 엔드투엔드 테스트는 보통 클라이언트 인터페이스—흔히 웹 브라우저나 휴대전화—를 기계적으로 조작하는 일을 수반한다. 나는 어떤 기술 리더에게도 클라이언트 조작 코드를 직접 짜라고 권하지 않겠다. 이건 특히나 까다로운 문제이고, Selenium, Cypress, Puppeteer 같은 내려받을 수 있는 도구가 대신 처리해 준다. 모바일에는 HeadSpin, Detox 같은 도구가 있다.

현실에서 엔드투엔드 테스트의 핵심 트레이드오프는 신뢰성이다. 적어도 이 글을 쓰는 시점에는, 신뢰할 만한 웹 엔드투엔드 테스트는 여전히 다소 잡기 어려운 목표다. 브라우저가 화면을 렌더링하는 방식의 특성상 경쟁 조건(race condition)이 의도치 않게 너무 쉽게 발생한다. 신뢰할 만한 웹 엔드투엔드 테스트 스위트를 구축하려면 상당한 주의와 디테일에 대한 집착, 그리고 꾸준한 유지·보수가 필요하다.

그래도 그 대가는 결코 작지 않다. 엔드투엔드 테스트 스위트로 사용자 대면 흐름에 대해 매우 높은 테스트 커버리지와 매우 높은 기능 확신을 만들어 낼 수 있기 때문이다. testim.io, rainforestqa.com을 비롯한 몇몇 회사가 이 문제를 AI와 머신러닝으로 풀려고 탐구하고 있다. 이런 솔루션은 이를테면 CSS 선택자의 존재 여부에 의존하는 대신 퍼지 시각 테스트(fuzzy visual testing)를 적용해 테스트 신뢰성을 높이려 한다. 바라건대 당신이 이 글을 읽을 무렵에는 기술 수준이 좀 더 나아가, 엔드투엔드 테스트의 가치 제안이 이 글을 쓰는 지금보다 한층 더 강해져 있을 것이다.

### 시각 회귀 테스트

시각 회귀 테스트(visual regression testing)는 렌더링된 화면의 차이(delta)를 구해 사용자 대면 애플리케이션의 결함을 잡아내려는 비교적 새로운 패러다임이다. 페이지 전체의 스크린샷을 찍는 것부터 개별 컴포넌트를 렌더링하는 것까지 다양한 단위로 이 작업을 수행하고 차이를 만들어 결함을 감지하는 프레임워크들이 있다. 명백한 단점은, 테스트 대상인 시각 컴포넌트에 의도적인 변경을 가할 때마다 테스트도 바꿔야 한다는 것이다.

다행히 이런 테스트 프레임워크는 비교 기준이 될 올바른 화면 모음을 간단하고 손쉽게 다시 만들어 주는 경우가 많은데, 이는 또 다른 함정을 연다. 테스트 기준을 덮어쓰는 도구가 손쉬운 탓에, 사실은 잘못된 시각 차이를 실수로 받아들이는 거짓 음성(false negative)을 내기가 매우 쉬워진다.

### 수동 테스트

수동 테스트는 이름 그대로 기계 코드가 아니라 사람이 수행한다. 코드를 테스트하는 사람을 기준으로, 이 범주는 다시 전문 테스터와 비전문 테스터로 나뉜다.

### 전문 수동 테스트

전문 수동 테스트는 사내 수동 테스트 팀이 맡는다. 정렬된 인센티브와 장기적 협업 관계 같은 사내 팀의 전통적 이점에 더해, 사내에 둠으로써 얻는 가치는 그 팀이 당신의 제품에 대한 전문성을 쌓고 사용자를 깊이 이해할 수 있다는 점이다. 덕분에 훈련되지 않았거나 익숙하지 않은 테스터라면 놓칠 결함을 그 팀은 짚어낸다. 수준 높은 테스트 팀은 소프트웨어 결함을 잡는 자원에 그치지 않고, 제품 차원에서도 값진 피드백을 보태 디자인의 비일관성을 찾아내고 무언가가 왜, 어떻게 그렇게 동작하는지 도발적인 질문을 던진다.

훌륭한 사내 수동 테스트 팀은 소프트웨어와 제품 품질 전반을 크게 끌어올린다.

전문 수동 테스트 팀은 제품 기능에 대한 상세한 테스트 계획을 작성하고, 그 계획을 쉽게 꺼내 재실행할 수 있는 형태로—이상적으로는 TestRail 같은 도구로—보관해야 한다. TestRail은 수동 테스트 계획의 완전한 스위트를 만들어 팀이 필요할 때 수동으로 다시 돌릴 수 있게 해 준다. 이런 도구의 또 다른 이점은 다른 개발·제품 도구와의 연동이다. 예를 들어 TestRail 실행을 Jira 에픽에 연결하면, 특정 기능을 출시하면서 수동·회귀 테스트를 몇 건 돌렸는지 보여 줄 수 있다. 이는 출시 체크리스트 항목으로 값질 뿐 아니라, 출시 후 발생한 결함을 회고할 때도 도움이 된다. 특정 기능을 내보내기 전에 어떤 수동 테스트를 돌렸는지 되짚어 보고, 빠져나간 결함을 잡을 수동 테스트를 추가할 수 있기 때문이다.

### 비전문 수동 테스트

비전문 수동 테스트는 흔히 크라우드소싱 테스트라고 불린다. Rainforest QA, Pay4Bugs, 99Tests, Testlio처럼 테스터를 모으는 전용 플랫폼이 여럿 있다. 이런 플랫폼의 가격 모델은 보통 검증된 버그 제출 건수에 따라 달라진다. 제품의 성격과 최적화하려는 결함 유형에 따라, 크라우드소싱 테스트는 매우 비용 효율적이고 손이 덜 가는 방식으로 제품 품질을 높여 준다.

## 반자동 테스트

반자동 테스트는 비교적 새로운 소프트웨어 테스트 범주다. 이 테스트는 비기술 인력—예컨대 전문 수동 테스트 팀—이 만들고, 그다음 완전 자동 환경 또는 감독이 붙은 환경에서 실행된다.

이런 테스트의 큰 함정은 신뢰성이다. 비기술 인력이 만들기 때문에 완전 자동화된 테스트만큼 정밀하지 않을 수 있고, 그래서 거짓 양성과 거짓 음성에 더 취약하다. 그렇지만 이 분야는 Rainforest QA, Testim처럼 신뢰성을 높이고 전체 비용을 낮추는 전략을 들고 해마다 새 회사와 도구가 등장하며 빠르게 진화하고 있다.
