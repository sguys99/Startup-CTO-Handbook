# 소스 관리

소스 관리에 관한 한 업계 표준은 Git이며, 거의 모든 회사가 Git을 쓴다. 다른 도구를 택하려면 정말 설득력 있는 이유가 있어야 한다. 지금 함께 일하는 팀원이든 앞으로 합류할 팀원이든 대부분은 이미 Git의 기본기를 갖추고 들어온다. Git을 쓰지 않으면 그들에게 굳이 새 대안을 익히게 만들어 불필요한 학습 부담을 지우는 셈이다.

클라우드 Git 호스팅 플랫폼은 GitLab, GitHub, Bitbucket 세 곳이 대표적이다. 이 셋이 시장의 대부분을 차지하므로, 여기서 벗어나려 할 때도 신중하게 따져 봐야 한다.

Git의 학습 곡선은 모양이 흥미롭다. 대부분은 어느 지점에서 정체기에 들어선다. 정상 경로(happy path)를 따라가는 웬만한 시나리오는 어설픈 이해만으로도 그럭저럭 넘어가기 때문이다. 그러다 일이 틀어진다. 개발자가 커밋을 날려 먹거나 머지를 엉망으로 만드는 순간이 온다. 이때 학습 곡선의 나머지 절반을 오르지 못한 탓에 좌절과 지연이 따른다. 나는 팀 리더라면 곡선의 후반부를 마저 오르는 데 공을 들이라고 권한다. Git을 오직 커맨드 라인에서만 써 보면서 실제로 무슨 일이 벌어지는지 익혀라. reflog, 인터랙티브 리베이스, bisect, 그리고 내장된 여러 머지 전략을 공부하라. 이 지식으로 무장하면 생산성을 갉아먹는 문제군 하나를 통째로 피하고 팀원도 Git 전문가로 키워 낼 수 있다.


### 동료 리뷰

전문가들은 대체로 모든 코드 변경에 탄탄한 동료 리뷰 절차를 두라고 권한다. (이 글을 쓰는 2023년 초 기준으로, 이 권고에 의문을 제기하거나 적어도 단서를 다는 흐름이 커지고 있다. 다음 절에서 다룬다.) 동료 리뷰는 대개 코드 호스팅 솔루션에 따라 풀 리퀘스트, 코드 리뷰, 머지 리퀘스트라 부르는 방식으로 이뤄진다. 코드 리뷰를 생산적이고 효율적으로 유지하기 위한 몇 가지 제안을 소개한다.

* 리뷰는 작게 유지하라! 코드 리뷰의 최대 크기를 정하라. 이를테면 파일 열 개, 200줄 정도가 적당하다. 그보다 큰 변경은 여러 개의 스택형/점진형 리뷰로 쪼개야 한다. (스택형 리뷰란 앞선 리뷰에 기반하거나 의존하는 코드 리뷰를 말한다. 완료되면 순서대로 머지되어 하나의 완전한 변경을 이룬다.)

* 코드 리뷰의 목표를 팀과 미리 합의하고 문화로 자리 잡게 하라. 코드 리뷰는 스타일이나 사소한 표기 시비를 가리는 자리가 아니다. 그건 자동 코드 포매터/린터와 정적 분석이 할 일이다. 코드 리뷰의 목적은 명료함을 확보하고, 아키텍처상의 우려를 짚고, 결함과 패턴 이탈을 잡아내고, 엣지 케이스를 기록하고, 비즈니스 규칙을 지키게 하는 데 있다.

* 작성자가 리뷰어의 일을 쉽게 만들도록 요구하라. 작성자는 변경 내용에 대한 설명, 관련 요구사항·티켓 링크, 그리고 (loom.com 같은 도구로 찍은) 코드와 코드의 정상 동작을 보여 주는 영상 워크스루를 함께 넣어야 한다.

* 코드 리뷰 작성자가 남에게 리뷰를 부탁하기 전에 스스로 한 번 리뷰하도록 권하라. 작성자가 독자를 안내하려고 적절한 위치에 남긴 코멘트 몇 줄이 많은 시간을 아껴 준다.

* 방해를 최소화하도록 리뷰 전용 시간/구간을 따로 떼어 두라.


### Ship, Show, Ask

흔히들 모든 코드 변경은 고객에게 출시되기 전 두 사람의 리뷰를 거쳐야 한다고 말한다. 다 그렇듯 여기에도 트레이드오프가 있다. 수동 코드 리뷰는 공짜가 아니며, 소프트웨어 품질을 보장해 주지도 않는다. 비용이 드는 만큼, 그 비용이 언제 가장 큰 수익으로 돌아오는지 따져 보고 코드 리뷰를 ROI가 가장 높은 상황에 쓰는 도구로 삼는 편이 좋다. 이 발상은 2021년 로언 윌세나흐(Rouan Wilsenach)의 블로그 글 〈Ship/Show/Ask〉(ctohb.com/ssa)로 널리 알려졌다.

코드 리뷰의 비용을 따져 보자. 코드 리뷰에서는 두 사람, 작성자(Author)와 리뷰어(Reviewer)가 여러 번의 컨텍스트 스위칭을 겪는다. 흔한 비동기 코드 리뷰 패턴은 다음과 같다.

**컨텍스트 스위칭 1:** 작성자가 프로젝트 1의 코딩을 멈추고 코드 리뷰를 준비한 뒤 리뷰어를 태그한다. 그러고 작성자는 프로젝트 2 작업을 시작한다.

**컨텍스트 스위칭 2:** 리뷰어가 알림을 받고 프로젝트 3 작업을 멈춘 뒤 프로젝트 1의 리뷰를 시작한다. 리뷰어는 작성자에게 피드백을 남기고 프로젝트 3 작업으로 돌아간다.

**컨텍스트 스위칭 3:** 작성자가 프로젝트 1에 대한 피드백 알림을 받고 프로젝트 2 작업을 멈춘 뒤 리뷰어의 코멘트를 처리한다. 그러고 작성자는 다시 프로젝트 2 작업으로 돌아간다.

**컨텍스트 스위칭 4:** 리뷰어가 프로젝트 3 작업을 멈춘다. 가장 좋은 경우라면 리뷰어가 이제 프로젝트 1의 변경에 만족해 코드 리뷰를 승인하고, 프로젝트 3 작업으로 돌아간다. 최악의 경우 작성자와 리뷰어는 컨텍스트 스위칭 3과 4를 여러 번 반복해야 한다.

**컨텍스트 스위칭 5:** 작성자가 승인 알림을 받고 프로젝트 2 작업을 멈춘 뒤 프로젝트 1을 머지하고, 다시 프로젝트 2 작업으로 돌아간다.

이런 컨텍스트 스위칭을 줄이는 방법이 없는 건 아니지만 그 방법에도 트레이드오프가 따른다. 흔한 대안은 모든 코드 리뷰를 동기식 페어 프로그래밍으로 처리하는 것이다. 그러나 이 전략은 컨텍스트 스위칭을 동기식 회의 시간과 맞바꿀 뿐이며, 그 회의 시간 역시 생산성을 끌어내린다. 어떻게 잘라 봐도 사람이 하는 코드 리뷰는 비싸다.

내가 제안하는 대안은 작업을 리스크 수준과 코드 리뷰로 기대할 수 있는 이득에 따라 분류하는 것이다. 분류 체계의 예를 들면 다음과 같다.

 사소한 변경 — 승인 불필요

- 카피/번역 문구 수정
- 사소한 UI 변경 (되도록 변경 결과를 보여 주는 시각 자료와 함께 제출)
- 테스트만 바꾸는 변경
- 아직 명시적으로 쓰이지 않거나 기능 토글 뒤에 비활성화된 신규 코드
- 어떤 고객이나 사용자도 접근할 수 없는 코드 (예: 숨겨진 페이지)

 경미한 변경 — 최소한의 리뷰 또는 사후 리뷰

- 테스트가 딸려 있고 기존 패턴·기능을 보강하는 코드 변경
- 실제 사용이 제한적이거나 전혀 없는 코드 (예: 미배포 제품)
- 신뢰할 만한 테스트로 정확성을 입증할 수 있는 리팩터링

 중대한 변경 — 사전에 꼼꼼한 리뷰

- 새로운 도구·프레임워크·패턴·아키텍처가 얽힌 모든 것
- 비중 있는 신규 기능
- 민감 데이터나 PII가 얽히거나 보안 태세에 영향을 줄 수 있는 모든 것

이 체계가 팀 전체의 효율을 높인다고 믿지만, 누구에게나 가능한 선택지는 아님을 인정한다. PCI나 SOC 2 같은 여러 컴플라이언스 체계는 100퍼센트 사람 코드 리뷰를 정책으로 요구한다. 그런 상황에서 할 수 있는 최선은 규정을 지키되, 컴플라이언스 프레임워크의 적용을 받지 않는 제품이나 기능 영역을 따로 떼어 내 더 정교하고 효율적인 절차를 실험해 보는 것이다.

### 브랜칭 모델

소스 관리의 브랜칭을 다루는 방법은 여러 가지지만, 업계 전반은 트렁크 기반 개발(trunk-based development)이라는 개념을 중심으로 힘을 모으고 있다. 현시점에서 가장 효과적이고 널리 쓰이는 패턴으로 보이므로 여기서는 이 방식을 다룬다. 다른 패턴을 진지하게 고려하고 있다면, 대안적 접근의 방법론과 모범 사례를 다룬 자료를 온라인에서 풍부하게 찾을 수 있다.

Git 브랜칭 모델을 유용한 그림과 함께 설명하는 블로그 글이 많은데, Reviewpad의 이 글(ctohb.com/branching)이 한 예다. 아래 설명이 잘 와닿지 않는다면 이런 글과 거기 딸린 시각 자료를 꼭 참고하길 권한다.

전통적인 브랜칭 모델, 이른바 GitFlow에는 main과 develop이라는 오래 유지되는 두 브랜치가 있다. 작업은 develop을 기반으로 한 기능 브랜치에서 이뤄지고, 릴리스마다 또 다른 릴리스 브랜치로 갈라졌다가 마지막에 main으로 다시 머지된다. 핫픽스는 main에서 진행하고, 추가 개발은 develop에서 이어 간다. 이 방식은 변경 하나가 프로덕션에 닿기까지 적어도 네 개의 브랜치를 거치며, 여러 브랜치를 동시에 관리해야 한다. 이런저런 이유로 GitFlow는 대체로 외면받게 되었고 더 이상 모범 사례로 여겨지지 않는다.

트렁크 기반 개발과, 그것을 조금 더 정교하게 다듬은 사촌 격인 GitHub Flow는 브랜치의 개수와 수명을 최소화하려는 소스 코드 관리 모델이다. GitHub Flow와 트렁크 기반 개발의 구체적 구현은 저마다 다르지만, 공통점은 이름이 무엇이든(이름은 그리 중요치 않다) 단일 브랜치가 존재한다는 것이다. 여기서는 그 브랜치를 production이라 부르겠다. production은 언제나 배포 가능한 상태다. 실제로 나는 production에 들어온 모든 커밋이 곧장 프로덕션에 배포될 수 있도록 자동화를 구성하라고 권한다. 작업은 production에서 갈라낸 기능 브랜치에서 진행하고, 그 기능 브랜치에서 리뷰한 뒤 준비가 되면 머지한다. 이게 전부다. 오래 유지되는 브랜치 하나와, 짧게 살다 사라지는(되도록 작은) 기능 브랜치 여러 개.

이 모델이 잘 굴러가려면 몇 가지 전제 조건이 필요하다.

* 기능 브랜치를 안전하게 머지할 수 있는지 확인해 주는, 탄탄한 테스트 묶음을 돌리는 지속적 통합(Continuous Integration).

* 브랜치를 빠르게 머지해 두고 기능은 사업상 적절한 시점에 나중에 배포·활성화할 수 있도록, 기능 토글을 쓰는 문화와 그 구현.

* 변경을 감지할 수 있는 탄탄한 프로덕션 모니터링.

* 무중단으로 프로덕션 환경에 코드 변경을 신속히 배포할 수 있는 능력. 마찬가지로, 장애가 발생했을 때 개별 변경을 신속히 되돌릴 수 있는 능력.

* 작고 수명이 짧은 기능 브랜치를 지키는 규율 있는 문화. 기능 브랜치가 커지고 오래 살아남아 다루기 힘들어지면 GitHub Flow 모델은 효율과 단순함을 잃는다. 3.3.5 기능 브랜치 환경에서 다뤘듯, 작은 커밋, 작은 브랜치, 작은 풀 리퀘스트가 생산성을 끌어올리는 핵심 동력이다.

### 오래 가는 브랜치 vs. 짧게 사는 브랜치

팀의 브랜치와 머지를 매끄럽게 유지하는 비결은 브랜치를 짧게 살리는 데 있다. 코드 머지와 얽힌 문제는 거의 다 브랜치가 너무 오래 열려 있거나 브랜치에 담긴 diff가 너무 커서 생긴다(ctohb.com/diffs). 일반적으로 수명이 짧은 브랜치는 며칠, 아무리 길어도 2주 안에 닫혀야 한다.

기능 하나를 반드시 단일 브랜치에서 구현할 필요는 없다는 점을 염두에 두라. 예를 들어 테스트만 담은 첫 브랜치를 만들어 리뷰·머지한 뒤 구현을 담은 브랜치를 이어 붙일 수 있다. 또는 본 애플리케이션과 연결되지 않은 구현을 먼저 만들어 리뷰·머지한 다음, 후속 브랜치에서 연결부와 테스트를 만들 수도 있다.

조금만 고민하고 연습하면 웬만한 구현은 독립적으로 머지할 수 있는 조각으로 나눌 수 있다. 이건 당신의 지도 아래 팀이 시간을 두고 길러 갈 수 있는 기술이다.

브랜치를 짧게 유지할 때의 이점은 다음과 같다.

* 다른 브랜치의 새 코드가 트렁크로 머지되는 시간 창을 좁혀, 코드 충돌의 여지를 줄인다. 작은 브랜치는 본래 충돌이 일어날 표면적도 작다.

* 기능 브랜치의 코드가 상대적으로 작게 유지되어 리뷰어가 읽기 쉬워지고, 망가질 여지도 줄어든다.

* 리뷰에서 더 빠른 피드백을 끌어내고, 기능을 구현해 가는 과정에서 더 일찍 방향을 바로잡을 수 있게 한다.

* 팀이 신뢰할 만한 지속적 통합 시스템을 갖추도록 부추긴다. 잦은 머지는 빌드/테스트 환경의 약점을 드러내, 시스템이 불안정하면 고통스럽게 만들고 그 시스템을 개선하도록 동기를 부여한다.
